# WINES.md — Wine Data Reference

This document describes how wine data is structured, stored, and interpreted across the Grape Expectations codebase. It is the canonical reference for anyone working on the wine inventory, the sommelier agent, or any external integration (e.g. MCP clients).

---

## Wine Record Fields

Defined in `js/types.ts` (`Wine` interface) and persisted in the `wines` Supabase table.

| TypeScript field | DB column | Type | Required | Description |
|---|---|---|---|---|
| `id` | `id` | UUID | yes | Primary key, generated by Supabase |
| `name` | `name` | string | yes | Wine name (e.g. "Château Margaux", "Cloudy Bay Sauvignon Blanc") |
| `winery` | `winery` | string | no | Producer/winery (e.g. "Domaine de la Romanée-Conti") |
| `vintage` | `vintage` | string | no | Year as a string — use `"NV"` for non-vintage wines |
| `type` | `type` | string | yes | One of the six wine types (see below) |
| `style` | `style` | string | no | Free-text style descriptor (e.g. "Full-bodied", "Crisp & mineral") |
| `country` | `country` | string | no | Producing country (e.g. "France", "Australia") |
| `region` | `region` | string | no | Wine region (e.g. "Bordeaux", "Barossa Valley") |
| `subRegion` | `sub_region` | string | no | Sub-region or appellation (e.g. "Pauillac", "McLaren Vale") |
| `price` | `price` | number \| null | no | Purchase price (SGD assumed) |
| `inventory` | `inventory` | integer | yes | Number of bottles currently in the cellar (default: 1) |
| `drinkFrom` | `drink_from` | integer \| null | no | First year the wine is recommended to drink |
| `drinkBy` | `drink_by` | integer \| null | no | Last year the wine is recommended to drink |
| `source` | `source` | string | no | How the wine was added (see Source Values below) |

> DB columns use `snake_case`; TypeScript fields use `camelCase`. Conversion happens in `mapDbWine()` in `js/utils.ts`.

## Source Values

The `source` field records how the wine entered the cellar.

| Value | Meaning |
|---|---|
| `manual` | User typed the wine details by hand |
| `photo_scan` | Claude vision extracted details from a label/invoice photo |
| `import` | Bulk imported (reserved for future use) |

## Drinking Window & Status

The drinking window is stored as two integer years (`drink_from`, `drink_by`) and interpreted at runtime by `getDrinkingStatus()` in `js/utils.ts`.

### Status values

| Status | Condition | Badge label | Badge colour |
|---|---|---|---|
| `prime` | `drink_from <= currentYear <= drink_by` | Prime | Green |
| `approaching_end` | In window, but `drink_by - currentYear <= 2` | Drink Soon | Amber |
| `past_peak` | `currentYear > drink_by` | Past Peak | Red |
| `too_young` | `currentYear < drink_from` | Too Young | Blue |
| `unknown` | Both `drink_from` and `drink_by` are null | Unknown | Grey |

### Priority order (for sorting)

When wines are sorted by drinking urgency: `past_peak → approaching_end → prime → too_young → unknown` (defined in `DRINKING_STATUS_PRIORITY` in `js/constants.ts`).

### Sommelier guidance per status

The sommelier system prompt uses these descriptions (`DRINKING_STATUS_DESCRIPTIONS` in `js/constants.ts`):

- **past_peak** — Mention this clearly if recommending; may still be enjoyable.
- **approaching_end** — Recommend urgently ("drink soon").
- **prime** — Currently at its best.
- **too_young** — Only recommend if the user specifically asks about aging potential.
- **unknown** — Treat as potentially drinkable.

### Auto-fill via Claude

When a wine is added without a drinking window, the app calls Claude to infer `drink_from` / `drink_by` from the wine name, winery, vintage, type, style, region, and country. See `App.tsx` around the `winesNeedingWindow` logic.

## How Tables Relate

```
wines  ──────────────────────────────────────────────────────┐
  └─ cellar_wine_id (optional FK)                            │
                                                             ▼
recommendation_sessions                           recommendation_feedback
  └─ recommendation_messages ──── message_id ───► recommendation_feedback
```

- `recommendation_feedback` links a thumbs up/down to a specific message and optionally to a `wines` row (when the recommended wine was already in the user's cellar).
- `in_cellar: true` on a feedback row means the recommendation was a cellar pick, not a "go buy this" suggestion — these carry different signals for future personalisation.

## Singapore Cuisine Pairings

Pairing logic lives in `js/localCuisine.ts` (not in the DB). It is injected into the sommelier system prompt at chat time. Key principles:

| Food profile | Pairing approach | Example dishes |
|---|---|---|
| Rich / fatty | High acidity cuts fat | Rendang, siu yuk, bak kut teh, char siu |
| Spicy | Off-dry aromatics tame heat; avoid tannic reds | Laksa, curry, sambal, chilli crab |
| Delicate seafood | Crisp, mineral whites | Teochew steamed fish, prawn dishes |
| Umami / braised / soy | Earthy reds or oxidative whites | Braised duck, lor bak, chicken rice |
| Wok hei / char | Body and smoky/toasty notes | Char kway teow, Hokkien mee, carrot cake |
| Sweet / caramelised | Ripe fruit or slight sweetness | Char siu, satay with peanut sauce |
| Tangy / tamarind | Bright acidity | Assam fish, rojak, tamarind prawns |
| Suckling pig (Teochew) | Fine bubbles + high acidity | Champagne, Blanc de Blancs, dry Riesling |

For multi-dish queries, the sommelier identifies the dominant pairing challenge and finds a wine that satisfies the most dishes without failing any. See `LOCAL_CUISINE_KNOWLEDGE` in `js/localCuisine.ts` for the full dish list.

## Cellar Analytics (Stats)

Computed by `computeStats()` in `js/utils.ts` from the full wine list. Stats surfaced in the UI and fed to the AI cellar health summary:

| Stat | Description |
|---|---|
| `totalBottles` | Sum of all `inventory` values |
| `uniqueWines` | Number of distinct wine rows |
| `avgPrice` | Average price across wines with a price set |
| `count2016` / `count2018` / `count2023` | Bottle count for those three vintages specifically |
| `modeCountry` | Most common country by bottle count |
| `modeStyle` | Most common style by bottle count |
| `drinkSoon` | Bottles in `approaching_end` status |
| `pastPeak` | Bottles in `past_peak` status |

## External Integration Notes (MCP / OpenClaw)

When querying `wines` via the Supabase MCP server:

- RLS is enforced by `user_id`. With a service role key, always filter by `user_id` explicitly — the MCP server bypasses RLS at the DB level.
- `drink_from` and `drink_by` are bare integers (years). Drinking window status must be computed client-side using the logic in [Drinking Window & Status](#drinking-window--status) above.
- `vintage` is stored as text — sort it numerically, and treat `"NV"` as a special case.
- Decrementing a bottle: `UPDATE wines SET inventory = inventory - 1, updated_at = NOW() WHERE id = '<id>'`. Do not set `inventory` below 0.
- The `WINES_JSON` comment block (see CLAUDE.md — AI Integration) is an implementation detail of the main app's feedback system. It appears in `recommendation_messages.content` for assistant turns; strip it before displaying raw message content.
